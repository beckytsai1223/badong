## Context

目前 `handlers.js` 持有 `ORGANIZER_IDS` 與 `isOrganizer()` 函式，負責在路由層過濾主辦人指令。`commands.js` 完全不知道使用者身份，所以所有回覆的提示區塊都是靜態字串，無法依身份產生不同內容。此外，點餐後菜單訊息隨聊天流向上移動，後來的成員必須往上滑才能找到菜單。

## Goals / Non-Goals

**Goals:**

- 點選餐點（`select_item` postback）或取消自己選餐（`/取消餐點`）後，自動 push 新菜單到群組底部
- 指令提示依主辦人/一般成員身份，各自只顯示自己能使用的指令
- 新增 `auth.js` 共用模組，讓 `commands.js` 可存取身份判斷邏輯

**Non-Goals:**

- 不改變任何現有指令的行為或路由邏輯
- 不新增資料庫欄位
- 不改變主辦人判斷的規則（仍從 `ORGANIZER_IDS` 環境變數讀取）

## Decisions

### 決策 1：抽出 `src/bot/auth.js` 共用模組

**選擇**：新建 `src/bot/auth.js`，內含 `ORGANIZER_IDS` 和 `isOrganizer(userId)`。`handlers.js` 和 `commands.js` 都 `require('./auth')`。

**理由**：`commands.js` 若直接 `require('./handlers')` 會造成循環依賴（handlers → commands → handlers），Node.js 會在其中一方取得空物件，導致 runtime 錯誤。獨立的 `auth.js` 無任何反向依賴，依賴圖是 DAG。

**備選方案**：在每個 command 函式簽名加入 `isOrganizer` boolean 參數 → 被拒。需修改所有呼叫點（handlers.js 的每個路由），且未來新增指令都需記得傳參數，維護成本高。

---

### 決策 2：HINT 改為函式 `hint(type, userId)`

**選擇**：將 `commands.js` 中的靜態 `HINT` 物件改為函式，呼叫時傳入 type 和 userId，內部呼叫 `isOrganizer(userId)` 決定顯示哪些指令。

```
hint('open', userId)
  → 主辦人：/統計、/確認下單、/取消餐點 <名字>、/取消訂單
  → 一般成員：點選菜單按鈕、/取消餐點
hint('confirmed', userId)
  → 主辦人：/收款狀態、/已收款 <名字>、/取消訂單
  → 一般成員：付款通知按鈕
hint('wizard', userId) → 無條件（只有主辦人會進入 wizard）
hint('none', userId) → 無條件（只有主辦人會觸發 none 狀態）
```

**理由**：呼叫點改動最小（`+ HINT.open` → `+ hint('open', userId)`），不需改變函式簽名。

---

### 決策 3：點餐後推菜單用 `pushMessage`，不用 `reply`

**選擇**：在 `selectItem` 和 `cancelMyItems` 中，`await replyText(...)` 後，再 `pushMessage(sourceId, buildMenuFlexMessage(...))` 把菜單推回群組。

**理由**：LINE API 每個 webhook event 只能呼叫一次 reply（消耗 replyToken）。若把菜單放進 reply，確認訊息就消失了。`pushMessage` 不受此限制，可在 reply 之後獨立發送。此模式已在 `confirmOrder` 使用。

`sourceId` 取自 `event.source.groupId || event.source.roomId || event.source.userId`（與 `confirmOrder` 相同）。`menuItems` 和 `order` 在這兩個函式中都已 fetch，直接複用，不額外查詢 DB。

      請逐一輸入菜單項目，格式：

      繼續輸入下一道，或傳送「/完成」結束。

        繼續輸入 <品名> <價格> 新增品項

## Risks / Trade-offs

**[風險] 推菜單增加 LINE API 呼叫次數**
→ 每次點餐多一次 `pushMessage` API 呼叫。10 人各點 2 次 = 20 次額外呼叫。LINE Messaging API 免費方案每月 200 則訊息；若群組活躍，可能接近上限。
→ 緩解：目前每次訂餐人數約 10 人，不會構成問題。未來若需要，可改為「每 N 次點餐才重推一次」，但目前不需要。

**[風險] 群組聊天室被菜單卡片洗版**
→ 每次點餐都出現一張菜單卡，10 人各點 3 份 = 30 張。視覺上可能雜亂。
→ 緩解：這是使用者明確要求的行為，且菜單卡本身就是功能性的（其他人可點選），不是純文字噪音。
